# LP解析とレポート作成スキル

このスキルは、Microsoft Clarityのヒートマップデータを活用してランディングページ（LP）を解析し、コンバージョン改善のための詳細レポートを作成します。

## 前提条件

- Microsoft Clarityアカウントへのアクセス権限
- 解析対象LPのClarityプロジェクトが設定済み
- LPのURL情報

## スキルの流れ

### ステップ1: Clarityダッシュボードへのアクセス

1. Clarityにログイン
2. 対象プロジェクトを選択
3. ダッシュボードでプロジェクト一覧を確認

### ステップ2: ヒートマップデータの収集

対象LPについて以下のデータを収集：

#### 2.1 スクロールヒートマップ
- ページのどこまでユーザーがスクロールしているか
- 各地点での離脱率
- スクリーンショットを保存

#### 2.2 アテンションヒートマップ
- ユーザーがどのセクションに注目しているか
- 各エリアの滞在時間
- スクリーンショットを保存

#### 2.3 クリックヒートマップ
- どの要素がクリックされているか
- デッドクリック（クリックされるが反応しない要素）
- デバイス別（モバイル/デスクトップ）の分析
- スクリーンショットを保存

#### 2.4 セッション録画の確認
- 実際のユーザー行動パターンを2-3件確認
- 離脱ポイントの特定
- 迷いや混乱の兆候を記録

### ステップ3: LPの構造確認

1. 対象LPのURLを訪問
2. 各セクションの構成を記録：
   - ヒーローセクション
   - 価値提案セクション
   - 価格・プラン
   - お客様の声
   - FAQ
   - CTAボタンの配置
3. ページ構造をドキュメント化

### ステップ4: データ分析と問題点の特定

収集したデータから以下を分析：

#### 4.1 クリティカルな問題（優先度：最高）
- 初動離脱率が30%以上
- コンバージョン阻害要因

#### 4.2 重要な問題（優先度：高）
- ユーザーの迷いや混乱
- 情報設計の問題

#### 4.3 改善推奨（優先度：中）
- UX最適化の余地
- デザイン改善点

### ステップ5: 修正案の立案

問題点に対して具体的な修正案を作成：

#### 5.1 即効性のある改善（Quick Wins）
- インパクト：高、実装難易度：低
- 1週間以内に実装可能な施策

#### 5.2 中期的な改善
- 2-4週間で実装する施策

#### 5.3 継続的改善
- A/Bテストや長期的な最適化

### ステップ6: レポートの作成

以下の構成でマークダウンレポートを作成：

```markdown
# [LP名] 解析レポートと修正案

## サマリー
- ターゲット
- コンバージョン定義
- 現状パフォーマンス

## Clarityデータ分析結果
- スクロールヒートマップの知見（画像付き）
- アテンションヒートマップの知見（画像付き）
- クリックヒートマップの知見（画像付き）
- セッション録画からの行動パターン

## LP構造の確認
- 現在のセクション構成

## 問題点の特定
- クリティカル、重要、改善推奨に分類

## 修正案（優先度順）
- 具体的な実装内容と期待効果

## 優先度マトリクス
- 表形式で整理

## 実装ロードマップ
- フェーズ1-3に分けた計画

## 次のステップ
```

### ステップ7: ファイル保存

レポートを2箇所に保存：

1. **アーティファクトディレクトリ**:
   ```
   /Users/ishikawasuguru/.gemini/antigravity/brain/<conversation-id>/[YYYYMMDD][lp_name].md
   ```

2. **プロジェクトディレクトリ**:
   ```
   /Users/ishikawasuguru/lp_heatmap/[YYYYMMDD][lp_name].md
   ```

## 出力ファイル命名規則

- フォーマット: `YYYYMMDD` + `lp_name` + `.md`
- 例: `20260210piste_lp.md`

## 重要なポイント

### データ収集時の注意点
- **デバイス別分析**: モバイルとデスクトップで行動が異なる場合がある
- **流入元の確認**: 広告経由とオーガニック流入でパターンが違う
- **期間の選定**: 直近3-7日間のデータが推奨

### レポート作成時の注意点
- **データに基づく**: 推測ではなく実際のヒートマップデータを根拠にする
- **具体的な提案**: 「改善すべき」ではなく「こう変更する」と明記
- **優先順位付け**: インパクトと実装難易度でマトリクス化

### 保存時の注意点
- **画像の埋め込み**: ヒートマップのスクリーンショットは絶対パスで参照
- **バックアップ**: 2箇所に保存することで紛失を防ぐ
- **命名の一貫性**: 日付を含めることで時系列管理が容易

7. レポートを作成し、以下に保存：

## トラブルシューティング

### Clarityでデータが表示されない
- プロジェクトが実際に作成されているか確認
- トラッキングコードが正しく設置されているか確認
- データ収集に最低24-48時間必要

### ヒートマップのスクリーンショットが見えない
- 画像の絶対パスが正しいか確認
- 画像ファイルが実際に存在するか確認

### レポートが長すぎる
- Quick Winsに絞って簡潔にまとめる
- 優先度の低い修正案は別ドキュメントに分離
